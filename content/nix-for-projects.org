#+TITLE: Why I Use Nix for Every Project
#+AUTHOR: Test Author
#+DATE: 2026-04-18

Every project I start gets a =flake.nix=. Not because I enjoy writing Nix (nobody does), but because it solves the "works on my machine" problem permanently.

** The problem

You clone a repo. The README says:

#+BEGIN_QUOTE
Prerequisites: OCaml >= 4.14, opam, dune, libgmp-dev, pkg-config...
#+END_QUOTE

Twenty minutes later you're debugging a version mismatch on a dependency of a dependency. The project worked yesterday. Today your system updated something.

** The solution

#+BEGIN_SRC nix
{
  inputs.nixpkgs.url = "github:nixos/nixpkgs";

  outputs = { nixpkgs, ... }:
    let pkgs = nixpkgs.legacyPackages.x86_64-linux;
    in {
      devShells.default = pkgs.mkShell {
        packages = [ pkgs.ocaml pkgs.dune_3 pkgs.gmp ];
      };
    };
}
#+END_SRC

Run ~nix develop~. Everything is there. Same versions today, tomorrow, on your colleague's machine, in CI. Done.

** What Nix gives you

1. *Reproducibility* --- pinned inputs mean identical environments
2. *Isolation* --- nothing leaks from your system into the build
3. *Composability* --- combine tools from different ecosystems trivially
4. *Rollbacks* --- broke something? Previous generations still exist

** The cost

Nix has a learning curve shaped like a cliff. The language is unusual, the error messages are unhelpful, and the documentation assumes you already understand it.

But once you've paid that cost, you never think about dependency management again. That's worth it.
